Booking Wizard Architecture

How should the multi-step booking flow be structured?

A

Single Component with Step State

One BookingWizard.jsx component manages a step state variable (1-5). Each step renders a different section inline. All wizard state lives in one place.

Pros

  • Simple — all state in one component
  • Easy to pass data between steps
  • No routing complexity

Cons

  • File grows large as steps get complex
  • No URL-based step navigation (can't bookmark/refresh mid-flow)
  • All step code loaded upfront
B

Orchestrator + Step Components

BookingWizard.jsx holds shared state and orchestrates flow. Each step is its own component: DogSelection, ServiceSelection, DatePicker, SlotPicker, BookingConfirmation. Wizard passes state down, steps call back up.

Pros

  • Each step is focused and testable
  • Wizard stays small (~200 lines)
  • Follows the pattern from BookingDetailModal decomposition in Phase 1
  • Easy to add/reorder steps later

Cons

  • Prop threading between orchestrator and steps
  • Still no URL-based navigation
C

Route-per-Step with Shared Context

Each step is a nested route (/customer/book/dogs, /customer/book/service, etc.). A React Context or useBookingWizard hook holds shared state across routes.

Pros

  • URL reflects current step
  • Browser back/forward works
  • Each step lazy-loadable

Cons

  • Most complex — context management, route guards, state persistence
  • Overkill for a 5-step wizard
  • Risk of state loss on page refresh

💡 Recommendation: Option B

The orchestrator + step components pattern is the sweet spot. It mirrors the BookingDetailModal decomposition you've already done in Phase 1, keeps files small and testable, and avoids the over-engineering of route-per-step for a flow that takes under 2 minutes to complete. The wizard is unlikely to be bookmarked mid-flow.
